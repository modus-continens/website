Modus

Star

Modus is a Datalog dialect for building OCI container images and containerised task automation. Modus has the following goals:

  • Maintainability: make build definitions maintainable by enabling modularity, code reuse and automatic dependency resolution.
  • Efficiency: make builds efficient by parallelisation and fine-grained caching; provide tools for optimising image size.
  • Expressiveness: enable the user to express complex build workflows in an intuitive way.
  • Simplicity: provide minimalistic syntax, well-defined, non-Turing-complete semantics, and static build graphs.

Intuitive Build Model

To be comprehensible and ergonomic, a domain-specific language has to be based on a formalism that naturally maps to the problem domain. This formalism has to be powerful enough to express important problems in the domain. However, the formalism should not be too powerful, because this would create issues with computability and clarity.

Modus models container builds as a specific kind of logic programming. The semantics of Modus naturally maps to Dockerfile/BuildKit's model of container image builds. Modus allows specifying complex build logic, while guaranteeing that build dependencies can always be efficiently resolved.

Dockerfile is a set of build stages; build stages are sequences of instructions: 

FROM ubuntu:20.04 AS base
RUN apt-get update && apt-get install -y python3.8 \
    && rm -rf /var/lib/apt/lists/*

FROM base AS app
COPY src src
RUN cd src && make install
Modusfile is a set of logical rules; rule bodies are expressions: 

base :-
    from(f"ubuntu:20.04"),
    run("apt-get update && apt-get install -y python3.8 \
         && rm -rf /var/lib/apt/lists/*").

app :-
    base,
    copy("src", "src"),
    run("cd src && make install").

Parameterised Builds

Container images are intrinsically parameterised. For example, the image python:3.9-slim-bullseye is parameterised with the Python version 3.9 and the distribution options slim and bullseye. Interactions between these parameters determine how the images are built. Modus explicitly handles parameters and their interactions.

In Dockerfile, build parameters are global variables; interactions between parameters cannot be expressed: 

ARG DISTR_VERSION=20.04
FROM ubuntu:${DISTR_VERSION} AS base
ARG PYTHON_VERSION=3.8
RUN apt-get update && apt-get install -y python${PYTHON_VERSION} \
    && rm -rf /var/lib/apt/lists/*

FROM base AS app
COPY src src
RUN cd src && make install
In Modusfile, parameters and their dependencies are explicit: 

base(distr_version, python_version) :-
    from(f"ubuntu:${distr_version}"),
    run(f"apt-get update && apt-get install -y python${python_version} \
          && rm -rf /var/lib/apt/lists/*").

app(distr_version, python_version) :-
    base(distr_version, python_version),
    copy("src", "src"),
    run("cd src && make install").

app :- app("20.04", "3.8").

Dependency Resolution

Image parameters affect the sequence of operations required to build the target image. Modus enables the user to describe complex interactions between image parameters as logical relations. It then uses a logic inference engine to resolve dependencies and statically compute the build graph.


base(distr_version, python_version) :-
    (distr_version = "16.04"; distr_version = "18.04"; distr_version = "20.04"),
    from(f"ubuntu:${distr_version}"),
    run(f"apt-get update && apt-get install -y python${python_version} \
          && rm -rf /var/lib/apt/lists/*").

# Ubuntu 14.04 requires adding third-party repositories to
# install recent versions of Python
base("14.04", python_version) :-
    from(f"ubuntu:${distr_version}"),
    run(f"apt-get install -y software-properties-common && \
          add-apt-repository ppa:deadsnakes/ppa && \
          apt-get update && \
          apt-get install -y python${python_version}",
          && rm -rf /var/lib/apt/lists/*").

app(distr_version, python_version) :-
    base(distr_version, python_version),
    copy("src", "src"),
    run("cd src && make install").

Standard Library

Modus provides a library of builtin predicates to handle common data structures in the build logic. For example, the predicate semver_geq checks if the left version is greater or equal to the right version according to SemVer specification.

base(distr_version, python_version) :-
    # The rule is applied if Ubuntu's version is greater or equal to 16.04
    semver_geq(distr_version, "16.04"),
    from(f"ubuntu:${distr_version}"),
    run(f"apt-get update && apt-get install -y python${python_version} \
          && rm -rf /var/lib/apt/lists/*").

Parallel Builds

Building container images can be time consuming. Thanks to the statically computed build graph, Modus aggressively parallelises build execution. It also allows building multiple images at the same time.

In this example, the build query for the previously introduced Modusfile resulted in the parallel build of two images: app("20.04", "3.8") and app("18.04", "3.8").


$ modus build . 'app(X, "3.8"), semver_geq(X, "18.04")'

OCI Image Properties and Build Instructions

Modus provides a library of operators, such as ::copy for multi-stage builds, that encapsulate build-specific instructions and manipulation of OCI image properties.

builder(go_version) :-
    from(f"golang:${go_version}")
        ::set_workdir("/go/src/github.com/alexellis/href-counter/"),
    run("go get -d -v golang.org/x/net/html"),
    copy("app.go", "."),
    # Executing the build command in a specific environment:
    run("go build -a -installsuffix cgo -o app .")
        ::in_env("CGO_ENABLED", "0")
        ::in_env("GOOS", "linux").

release :-
    (
        from("alpine:latest")::set_workdir("/root"),
        run("apk --no-cache add ca-certificates"),
        # Copying files from another image for multi-stage build:
        builder("1.16")::copy("/go/src/github.com/alexellis/href-counter/app", ".")
    )::set_entrypoint("./app").

Optimising Image Size

Container images often include redundant files in their layers, which slows down their transfer through the network and compromises security by increasing the attack surface. By allowing the user to encode precise dependencies, such as the software packages to install or files to copy into the image, Modus helps to optimise the image size. Besides, Modus provides additional tools to reduce the image size such as the operator ::merge that merges several layers into one.


app(optimisation) :-
    from("gcc:latest"),
    # The following code will produce a single layer,
    # therefore the directory "src" will not be in the resulting image:
    (
        copy("src", "."),
        (optimisation = "on", run("cd src; make install-opt");
         optimisation = "off", run("cd src; make install")),
        run("rm -rf src")
    )::merge.

Modularity & Code Reuse

Just like any code, container build scripts evolve and require maintenance. Modus supports code evolution and maintenance by providing tools for modularity and code reuse. For example, Modus allows users to define their own commands, such as layer building functions or logical predicates, to abstract reusable build workflows.


# User-defined layer predicate for installing libraries
install(lib, version) :-
    run(f"wget https://example.com/releases/${lib}-v${version}.tar.gz && \
          tar xf ${lib}-v${version}.tar.gz && \
          mv ${lib}-v${version}/ /build"),
    run("cd /build && make install"),
    run(f"rm ${lib}-v${version}.tar.gz && \
          rm -rf /build").

app :-
    from("gcc:latest"),
    (
        install("liba", "1.3.5"),
        install("libb", "4.1")
    )::merge.
